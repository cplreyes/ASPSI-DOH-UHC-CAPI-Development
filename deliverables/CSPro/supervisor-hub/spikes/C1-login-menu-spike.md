# C1 spike — login/menu app on CSPro 8

**Status:** IN PROGRESS (build started 2026-06-24). Verdict recorded at bottom.

Phase-2 "full pivot to menu + hub" gating spike. Confirms Khurshid's 7.7-era
login+menu pattern still works on **CSPro 8 + itel**: `loadcase` auth against an
external roster, `savesetting`/`loadsetting` role handoff, and **`execpff` chain-launch**.

> **API grounding (2026-06-24, csprousers.org help):** the chain-launch is
> **`execpff(pff_filename | pff_object, flags)`** (returns 1/0; the `stop` flag stops the
> current app and launches the target; on mobile it queues the `.pff` to launch when the
> current app exits). It accepts a **static `.pff` filename string** directly — so this
> spike uses a pre-authored `.pff` + `execpff("…pff", stop)`, NOT the plan's sketched
> `PFF.setProperty()/.execute()` object API (the `pff_object` help page 404s and `.execute()`
> is not the documented method — `execpff` is). `savesetting(attr, value)` / `loadsetting(attr)`
> persist on-device across apps (confirmed). `loadcase(DICT, key)` is already proven in-repo
> (F1/F3/F4 PSGC cascades) — used here with the `<> 0` idiom.

## PASS criteria (ALL must hold)

1. A minimal **LoginApp** authenticates a username/password against an external roster
   dictionary via `loadcase()` (correct creds accept; wrong password → "Incorrect password."
   + re-enter; unknown username → "Unknown username." + re-enter).
2. On success, `savesetting("hub_role", …)` persists, and a separate **MenuApp** reads it via
   `loadsetting("hub_role")` after launch (the role is shown on the menu screen).
3. The LoginApp **chain-launches MenuApp via `execpff("MenuApp.pff", stop)`**, and the
   MenuApp chain-launches a target instrument `.pff` (e.g. `PatientSurvey.pff`) the same way.
4. All apps **compile in CSPro 8 Designer** (lenient) AND **publish** (strict packager) AND
   **run on the itel/emulator** (not just compile).

## FAIL → fallback

If chain-launch or `savesetting`/`loadsetting` does not work on CSPro 8 / itel, **C1 = FAIL**.
Phase B then skips the app-login (plan Tasks 3–5); roles are distinguished by **CSWeb account
only** (enumerator runs the instrument directly as today; supervisor uses the `supervisor-qa`
account + the hub relay/QA). Record exactly which CSPro 8 API differs from Khurshid's 7.7 example.

## Verdict

**`C1 = PASS` — device-confirmed on the itel (`091945939L000716`) 2026-06-24.** The Khurshid
login + role-handoff + chain-launch pattern WORKS on CSPro 8 + CSEntry. One packaging caveat
for Phase-B (instrument launch must be co-bundled — not an API blocker).

### Device run (deployed to production CSWeb as ONE LoginApp bundle, per Carl's call)

CSEntry → Add Application → CSWeb → **LoginApp** installed (MenuApp + UserRoster rode inside the
bundle; CSEntry registered both LoginApp and MenuApp as entry apps). Then:

1. **loadcase auth (criterion 1) ✅** — opened LoginApp → entered username `fs-01` (case key) →
   password `changeme-fs`. The `LOGIN_PASSWORD` postproc's `loadcase(USER_ROSTER_DICT, LOGIN_USERNAME)`
   matched and the password check passed (a wrong/unknown cred would have `reenter`-ed on the field;
   the positive path proves the loadcase + comparison branch — see `dev_09`/`dev_10`).
2. **savesetting → loadsetting cross-app handoff (criterion 2) ✅** — after auth, MenuApp opened and
   its case-tree showed **"Logged-in role: supervisor"**: `loadsetting("hub_role")` in MenuApp read
   exactly what LoginApp's `savesetting("hub_role", "supervisor")` wrote, ACROSS two separate apps
   (`dev_12`). The id field auto-set + skipped (`noinput`); `ROLE_SHOWN` showed read-only (`protect`).
3. **execpff chain-launch, Login→Menu (criterion 3a) ✅** — advancing past the password
   `execpff("MenuApp.pff", stop)` ended LoginApp and launched MenuApp (title bar "MenuApp - 0 Cases",
   `dev_11`). The mobile "queue on app-exit" behaviour worked transparently.
4. **execpff chain-launch, Menu→instrument (criterion 3b) ⚠️ PACKAGING CAVEAT** — selecting
   "F3 — Patient Survey" ran `execpff("PatientSurvey.pff", stop)` but **did not** launch PatientSurvey;
   MenuApp exited to the CSEntry app list (`dev_13`). Cause: PatientSurvey is installed as its OWN app
   in a SEPARATE folder, so the bare relative name `PatientSurvey.pff` doesn't resolve from MenuApp's
   folder. **execpff is NOT at fault** (3a proved it works when the target `.pff` is co-located).
   **Phase-B fix:** ship ONE bundle containing LoginApp + MenuApp + the instruments + their launch
   `.pff`s (Khurshid's all-in-one-bundle pattern), OR reference the instrument by its on-device path.
   This is a build/packaging task, not an API blocker.

### Net

The make-or-break — *does Khurshid's CSPro-7.7 login+menu+chain pattern work on CSPro 8 + CSEntry?*
— is **YES**: `loadcase` auth, `savesetting`/`loadsetting` role handoff, and `execpff` chain-launch
all work on-device. Phase-B Tasks 3–5 (roster, login app, menu app) are **UNBLOCKED**; the one design
note is to co-bundle the instrument launchers so Menu→instrument resolves. Negative-auth path
(wrong/unknown cred → reenter) is by-construction (same `loadcase`/`errmsg`/`reenter` branch) and
not separately exercised on-device.
