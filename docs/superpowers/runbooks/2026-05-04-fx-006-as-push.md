# FX-006 — Push `F2_AUDIT_COLUMNS` schema fix to staging Apps Script

> Unblocks the Audit-tab Actor / Resource / Detail columns on the staging Admin Portal. The Schema.js fix shipped in commit `3bd62dd` (extended `F2_AUDIT_COLUMNS` with the 7 admin-context columns the AP1–AP4 migration added). The compiled `dist/Code.gs` is regenerated and ready to push; only the Apps-Script-side paste is left.

## Step 1 — Rebuild dist (~30 sec, idempotent)

If you've made any backend changes since 2026-05-03, rebuild. Otherwise the existing `dist/Code.gs` already has the fix.

```powershell
Set-Location deliverables/F2/PWA/backend
npm run build
```

Output: `dist/Code.gs` + `dist/appsscript.json` + `dist/Admin.html`. The build is idempotent — running it again on a clean tree produces byte-identical output.

> Sanity check: `Get-Content dist/Code.gs | Select-String -Pattern 'F2_AUDIT_COLUMNS' | Select-Object -First 3` should show the extended column list — `actor`, `resource`, `detail`, `request_id`, `client_submission_id`, `client_ip`, `user_agent` should all appear.

## Step 2 — Open the staging Apps Script project (~1 min)

1. Open https://script.google.com while signed in as `aspsi.doh.uhc.survey2.data@gmail.com`.
2. Open the project named **`F2-PWA-Backend-Staging`** (set up per `docs/superpowers/runbooks/2026-05-01-ap0-apps-script-staging-setup.md` Part 2).
3. Verify you're on the staging project, not production: top-left should say `F2-PWA-Backend-Staging`. **Do NOT push to `F2-PWA-Backend` (production)** — production-side AS deploy rides with E4-APRT-040 v2.0.0 in Sprint 005.

## Step 3 — Paste the new Code.gs (~1 min)

1. In the staging AS Editor, click `Code.gs` in the left file tree.
2. Select all contents (`Ctrl+A`) and delete.
3. Open `dist/Code.gs` from this repo locally, copy all contents, paste into the AS Editor.
4. Click the **Save** floppy icon (or `Ctrl+S`). Wait for "Saved" to appear next to the project name.

> Subtle: Code.gs in `dist/` is a single-file bundle that includes Schema.js + AdminAudit.js + AdminHCWs.js + Setup.js + Migrations.js + Code.js — the build flattens them. You're pasting one file, not many.

## Step 4 — Deploy a new version to the existing Web App (~2 min)

The staging Web App URL is already wired up to the worker's `APPS_SCRIPT_URL` secret. You're not creating a NEW deployment — you're publishing a new version of the EXISTING deployment so the URL stays the same.

1. Editor → **Deploy** → **Manage deployments**.
2. The list shows your existing staging Web App (URL ending in `/exec`). Click the **pencil icon** to edit it.
3. **Version:** dropdown → **New version**.
4. **Description:** `FX-006 — extend F2_AUDIT_COLUMNS for admin-context fields`.
5. Leave Execute as / Who has access untouched.
6. Click **Deploy**.
7. Confirmation modal shows the same `/exec` URL as before — that's correct (URL stable across versions). Close the modal.

## Step 5 — Verify (~30 sec)

Hit the staging admin Audit endpoint via the Worker (which calls AS) — easiest path is to log in to the portal and refresh the Audit tab:

1. Open `https://staging.f2-pwa.pages.dev/admin/login`. Login as `carl_admin` / `100%SetupMe!`.
2. Click **Data** → **Audit** sub-tab.
3. The audit log table should now populate **ACTOR** column (`carl_admin` for admin_login / admin_revoke_sessions rows) and **RESOURCE** + **DETAIL** where applicable. If columns still show `—`, the AS deployment didn't take — re-check Step 4 (did you click the pencil to edit, or did you create a new deployment? new deployment changes the URL and breaks the worker wire).

> Alternate verify (no portal): `curl.exe -X POST "$STAGING_AS_URL?action=admin_audit_list" -H "Content-Type: application/json" -d "{\"hmac_token\":\"...\",\"args\":{\"limit\":3}}"` — but signing the HMAC by hand is fiddly, the portal verify is faster.

## What NOT to do

- **Don't push to production** (`F2-PWA-Backend` without `-Staging`). Prod AS deploy is part of E4-APRT-040 in Sprint 005 with v2.0.0.
- **Don't create a new deployment** in Step 4. Use the pencil-edit-existing-deployment path so the `/exec` URL stays stable. Creating a new deployment generates a new URL, the worker `APPS_SCRIPT_URL` would still point at the old URL, and the new version wouldn't be reachable.
- **Don't run `setupBackend()` or `rotateSecret()`** unless you actually need to reset state — these are first-deploy / break-glass operations, not for routine pushes.
- **Don't paste only changed lines.** Always replace the whole `Code.gs` contents — partial pastes leave stale code in the rest of the file.
