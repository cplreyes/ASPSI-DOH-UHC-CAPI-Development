# Source — Survey Manual Appendix F (Patient Listing Form)

Minimal **paper template** for the Patient Listing Form, circulated by [[1_Projects/ASPSI-DOH-CAPI-CSPro-Development/wiki/entities/Juvy Chavez-Rocamora|Kidd]] alongside the [[Source - Survey Manual Working File (2026-05-06 Kidd)|Working File]] on 2026-05-06. ~208 lines.

## Document structure

- **Header block (L1–L43):** five metadata fields — Region, Province, Site (UHC IS / non-UHC IS checkbox), Facility Name, Type of respondents (Outpatient / Inpatient checkbox).
- **Listing table (L45–L208):** 25 numbered rows × 2 columns — **Contact Number** + **Family Name (Optional)**.

That is the entire form.

## What's missing

> [!warning] Significant drift from the F3b protocol
> The form captures **only contact number + optional family name** for up to 25 patients. The [[1_Projects/ASPSI-DOH-CAPI-CSPro-Development/wiki/sources/Source - Annex F3b Patient Listing Protocol|F3b Patient Listing Protocol]] (already in wiki, ingested earlier) and the [[Source - DOH Survey Protocol V2 (30 April)|Protocol V2]] both describe a much richer CSPro-driven workflow including:
>
> - **Randomization metadata** — CSPro-generated random interval (1–10 min) per approach, sequence number, listing date/time, enumerator ID
> - **Eligibility flags** — refusal indicator, exclusion reason (deceased, ICU/critical, pediatric without guardian)
> - **Survey tagging** — exit survey vs follow-up household survey assignment based on distance/transport/contact info
> - **Contextual fields** — patient age/sex, distance from home, mode of transport, diagnosis/chief complaint
> - **Linking key** — listing_id or facility-date-sequence for join to F3 patient survey
>
> **None of these appear in Appendix F.** The form reads as a paper backup roster only — adequate when the CSPro listing app is offline / unavailable, but not the full specification.

## CAPI implementation implications

A separate **listing dictionary** in CSPro is needed to hold the operational metadata that the paper form omits:
- Randomization interval (generated by CSPro)
- Listing sequence + date/time + enumerator ID
- Refusal / exclusion flags
- Survey-tag assignment (exit vs follow-up household)
- Distance from home / mode of transport (carry forward to F3 if not re-asked)

The F3 patient survey can reference the listing ID to inherit contact + context info, but **the paper form provides no listing_id field** — so reconciliation paper→CSPro requires a manual key (likely facility-date-sequence assigned at ingest time).

## Open questions

1. **CSV/CSPro handshake** — How does contact number link from paper form to F3 survey when the paper backup is used? No listing_id or unique key on the form.
2. **Inpatient vs outpatient** — header checkbox selects respondent type but the listing table is identical. F3b protocol specifies *separate* protocols (discharge station vs OPD registration). Which fields should differ between the two paper variants?
3. **Randomization not captured** — the form records *who was selected* but not *how* (random interval, sequence). If CSPro is offline, how is randomization logged for audit?
4. **Survey-tag field absent** — F3b protocol describes "exit survey" vs "follow-up household survey" tagging; Appendix F has no such field.
5. **Distance / transport** — listing-day captures per F3b protocol; missing from Appendix F. Re-asked in F3, or expected on Appendix F?
6. **Why "Optional" on Family Name** — if Contact Number alone doesn't uniquely identify a patient for cross-referencing, this is a data-integrity gap.

## Verdict

Appendix F is a **minimal paper fallback**, not the full F3b specification. The protocol (Carl's [[1_Projects/ASPSI-DOH-CAPI-CSPro-Development/wiki/sources/Source - Annex F3b Patient Listing Protocol|F3b]]) and the form (this Appendix F) should be **reconciled** — Appendix F needs metadata fields for CSPro-offline scenarios, and the architecture should clarify the listing-to-F3 linking mechanism.
